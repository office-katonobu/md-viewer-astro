### 一括印刷専用ページの設計方針

個別の記事ページとは完全に切り離し、`/print-all` という専用ルートを作成することで、印刷時のレイアウトを最適化します。

#### 1. ページ構造 (src/pages/print-all.astro)

全ドキュメントを1ページに読み込み、シームレスなドキュメントセットを生成します。

* **目次エリア**: 全ドキュメントのタイトルをリスト化し、各ドキュメントのID（`id={doc.slug}`）へジャンプできるようにします。
* **コンテンツエリア**: `getCollection` で取得したデータをループし、各ドキュメントを `<article>` タグで包んで展開します。
* **タイトル付与**: 各記事の先頭に `{セットタイトル} - {各ドキュメントタイトル}` を動的に出力します。

#### 2. 印刷用レイアウト制御（CSSの要点）

`@media print` を活用して、紙面としての完成度を高めます。

* **自動改ページ**: 各ドキュメントの後ろで確実に改ページを発生させます。
```css
@media print {
  .print-section { page-break-after: always; }
}

```


* **不要要素の排除**: Web閲覧用のサイドバーやフッターなど、印刷時に不要なUIを `display: none !important;` で完全に消去します。
* **デザイン調整**: 画面用レイアウトを無視し、紙の幅（100%）を最大限に活用するスタイルへ上書きします。

### 実装の全体像イメージ

1. **アクセス**: `/print-all` にアクセス。
2. **目次ページ**: 最初の一枚にドキュメント一覧が表示される（ここも1ページとして認識される）。
3. **コンテンツ**: 2ページ目以降、各ドキュメントが独立したセクションとして続く。
4. **印刷開始**: ブラウザの印刷メニューが開き、PDF保存やプリンタ出力を行う。

### 今後の開発ステップ

1. **ページ作成**: `src/pages/print-all.astro` を作成し、目次とコンテンツ展開ロジックを記述する。
2. **スタイル適用**: `global.css` を読み込み、印刷時に不要な要素を隠すためのクラス（`.no-print` 等）を共通化する。
3. **リンク設置**: `index.md` に「ドキュメント一括印刷（PDF出力用）」というリンクを配置し、完成です。

### 一括印刷システム要件・実装整理表

本システムは、膨大なドキュメントを「Webでの連続閲覧」と「紙面・PDFへの印刷」の両方に最適化することを目的としています。

---

#### 1. システム全体および機能要件

| 要求事項 | 実現方法（実装上の対応） |
| --- | --- |
| **全ドキュメント統合** | `getCollection('docs')` を使用して全ファイルを一括取得し、`map()` でページ内に連続して展開しました。 |
| **目次からのナビゲーション** | 各ドキュメントにファイル名由来のユニークなID（`id={doc.id}`）を付与し、目次からそのIDへ直接飛べる（`#id`）アンカー構造を構築しました。 |
| **印刷時のレイアウト制御** | `@media print` CSSを定義し、各ドキュメント（`.print-section`）の末尾に `page-break-after: always;` を適用することで、物理的な改ページを強制しました。 |
| **相対リンクの解決** | `LinkFixer.astro` コンポーネントを配置し、JavaScriptでMarkdown内の `.md` リンクを検知。リンク先を該当するページ内アンカー（`#id`）へ書き換えるロジックを実装しました。 |
| **画面・印刷のUI出し分け** | `.no-print`（印刷時非表示）や `.print-only`（印刷時のみ表示）クラスをCSSで定義。目次への戻りリンクなどを、閲覧環境に応じて適切に表示・非表示に制御しています。 |

---

#### 2. 実装上の技術的要求事項と解決策

システム開発の過程で浮上した技術課題に対し、以下の解決策を採用しました。

* **Astro v7系仕様への最適化**
* **課題**: 従来の手法（`doc.render()`）が動作しない。
* **解決**: Astro v7の標準である `doc.rendered.html` を利用し、`<div set:html={...} />` で直接HTMLをレンダリングする方式に改めました。


* **動的ルーティングエラーの解消**
* **課題**: コンポーネントを `src/pages/` に置いたことで `getStaticPaths` のエラーが発生。
* **解決**: コンポーネントを `src/components/` ディレクトリへ適切に分離し、ページ生成ロジックと部品を明確に区別しました。


* **画面閲覧時の連続表示の冗長性緩和**
* **課題**: ドキュメントが連続するとどこで区切れているか分かりづらく、フッターの戻りリンクが過剰に見える。
* **解決**:
1. 画面表示時のみ、`.print-section + .print-section` セレクタを用いて、記事間に明確な境界線（border）と余白を付与。
2. フッターの「目次に戻る」リンクは画面上では `display: none` に設定し、印刷時のみ表示するようにCSS制御しました。




* **リンク書き換えの自動化**
* **課題**: 手動でリンクを書き換えるのは非効率でメンテナンス性が低い。
* **解決**: `LinkFixer.astro` に辞書（`urlRegistry`）を持たせ、ページ読み込み時にDOMを操作して自動でリンクを補正する仕組み（クライアントサイドスクリプト）を導入しました。

### まとめ：本システムの強み

このシステムは、「コンテンツデータは一元管理し、表示用のCSSとJavaScriptで閲覧モードと印刷モードの役割を切り替える」という設計思想に基づいています。

1. **メンテナンス性**: Markdownを修正するだけで、Webページと印刷用PDFの両方に内容が反映されます。
2. **一貫性**: `LinkFixer` により、Markdown内のリンクがどのページから参照されても、確実にシステム内の正しい場所へ誘導されます。
3. **ユーザー体験**: 印刷物としては「紙の資料」としての体裁（改ページ・目次戻りリンク）を保ちつつ、Webサイトとしては「連続して読めるドキュメント」としての使い勝手を両立させています。
