# 📑 ドキュメントシステム構築プロジェクト：フェーズ2 最終報告書

本ドキュメントは、Astro（Content Collections）を用いた社内ドキュメントシステムにおける、構造改善、リンク補正、UI自動連動、および環境変数による運用制御の実装内容をまとめたものである。

---

## 1. 要求事項 (Requirements)

フェーズ2におけるシステム改善および機能追加の要求事項は以下の通り。

1. **ドキュメントツリーの並び順最適化**
サイドバー内のフォルダ・ファイル一覧において、視認性を向上させるため、同一階層内では常に「ファイルを先、フォルダを後」の順で並び替えて描画すること。
2. **サイドバーの初期全閉じ化とURL連動開閉**
初期表示（TOPページ等）ではサイドバーのフォルダをすべて閉じた状態とする。特定のページにアクセス、または遷移した際は、ブラウザのURL階層を解析して「該当する親フォルダ」だけを全自動でピンポイント展開（開状態に）すること。
3. **現在地のメニューハイライト**
現在表示しているドキュメントに該当するサイドバーのメニュー項目を、視覚的（背景色：青）に強調表示すること。
4. **本文内外部リンクの動的抽出とメニュー連携**
表示中のMarkdown本文内に記述されている「本物の外部リンク（`http(s)://`）」をシステムが自動で検知・抽出し、サイドバー最下部の「外部リンク」エリアへリアルタイムにリストアップすること。
5. **外部リンクの一括別タブ（`target="_blank"`）化**
サイドバーメニューからのクリック時だけでなく、Markdown本文中にあるすべての外部リンクをクリックした際も、社内ドキュメント画面を維持するために一括して自動で別タブ（新規ウインドウ）で開かせること。
6. **環境変数によるデバッグ表示のON/OFF制御**
開発・検証時に使用したシステム内部のステータスパネルを、本番環境や一般公開時には設定ひとつで非表示に切り替えられる運用設計とすること。

## 2. 実装方針 (Architecture & Implementation Policy)

要求事項を美しく、かつ最小限のファイル構成で解決するため、以下の割り当てで実装を分担・統合する方針をとった。

* **サーバーサイド（Astro/Node.js）処理:** ビルド（またはリクエスト）時に、全ドキュメントの「物理配置（ファイル名）」と「Astroが生成する正規URL」の完全なペア対応表（以下、**正解URLレジストリ**と呼ぶ）を構築する。また、Markdown本文の生のテキスト（`entry.body`）を正規表現スキャンし、外部リンクデータを事前に抽出してコンポーネント間で受け渡す。
* **クライアントサイド（HTML/JavaScript）処理:** ブラウザにHTMLがレンダリングされた直後、DOMを動的に操作する。
1. サーバーから引き継いだ「正解URLレジストリ」を用いて内部リンクを100%安全な絶対パスへ書き換える。
2. URL構造からアクティブな階層を特定してサイドバーの `<details>` タグを制御する。
3. 本文内の外部リンクに別タブ属性を一括注入する。

## 3. 技術的な問題点と最終的に選択した技術的方式 (Technical Challenges & Solutions)

### 🔴 直面した問題点：ブラウザによる相対パス計算の破綻

当初、Markdown内の相対パスリンク（例：`../records/ファイル名.md`）を解決するため、現在のブラウザのURL（`window.location.pathname`）をベースに階層移動（`..` による配列のポップ処理など）をフロントエンドで計算し、絶対パスに変換する方式を採用していた。

しかし、以下の要因によってクライアントサイドでの純粋な相対計算は容易に破綻することが判明した。

1. 日本語ディレクトリ名・ファイル名が混在することで、ブラウザによってURLが「生の日本語（デコード）」と「％の暗号（エンコード）」の間で不安定に行き来する。
2. リンク元とリンク先の階層構造の深さが異なる場合、現在のURLをベースにした相対階層の足し引き計算のロジックが極めて複雑化し、バグの温床となる。

### 🟢 最終的に選択した技術的方式：「絶対安全な辞書検索（レジストリマッチ）方式」

現在のページ位置から相対的に計算するのを一切やめ、Astroがシステム内部で把握している「全ドキュメントの正解URL対応表」を事前に作成し、フロントエンドへ受け渡してピンポイント検索する方式にパラダイムシフトした。

* **ステップ1：Astro側での辞書生成（サーバーサイド）**
`getCollection('docs')` を用いて、ビルド時に全ファイルの「生のファイル名」と「Astroが公式にルーティングするURLパス」を紐づけたオブジェクト配列（レジストリ）を自動生成し、非表示のデータ属性（`data-registry`）にJSONとして埋め込む。
* **ステップ2：ブラウザ側での完全一致解決（クライアントサイド）**
ブラウザ上のJavaScriptは、ページ内の各リンクをスキャンする際、そのリンク先文字列（またはリンクテキスト）をデコードし、レジストリ（辞書）内に存在するファイル名と部分一致・完全一致するかどうかだけを判定する。
一致した場合、Astroが保証する正規の絶対URLパス（例：`/records/%E5%81%9C...`）でリンクの `href` 属性を上書き強制補正する。

**【この方式のメリット】**
Markdown内でどのような相対階層（`../` や `./`）でリンクが記述されていようとも、**「最終的な目的地のファイル名」さえ合致していれば100%確実に正しいURLへ誘導できる**ため、ルーティング上の404 NotFoundエラーを永久に、根底からシャットアウトすることが可能となった。

## 4. デバッグ向け機能概要 (Debug Tooling Overview)

開発環境や、今後のMarkdownデータインポートフェーズにおける「リンク切れ・タイポ検証」を極めて効率的に行うため、**システム内部認識ステータス（検証パネル）** を独自に開発し、画面最上部に統合した。

```
+-----------------------------------------------------------------------------------------+
| ⚙️ システム内部認識ステータス (Debug Mode)                                                  |
|  - 解析中のファイルID（物理配置）: templates/災害時対応テンプレ.md                          |
|  - Astroシステムが認識している現在のURLパス: /templates/災害時対応テンプレ                    |
|  - 自動検出された外部リンク数: 2 件                                                       |
| 📋 ページ内Markdownリンクのリアルタイム補正ログ                                             |
|  [🎯 辞書一致成功] ➔ /records/%E5%81%9C... (元の記述: ../records/停電時業務継続記録.md)    |
|  [ℹ️ 外部リンク補正: 別タブ化 OK] ➔ "https://www.jipdec.or.jp/..."                       |
+-----------------------------------------------------------------------------------------+

```

### パネルの主な機能

1. **物理配置の可視化:** Astroが今どのファイルをベースにHTMLを作っているか（ファイルID）を生データで表示。
2. **リアルタイム補正ログ:** スクリプトがページ内の全リンクをスキャンした際の結果を1件ずつログ出力。
* 内部リンクの補正に成功した場合は **`🎯 辞書一致成功`**（緑色）
* ファイル名が辞書に見つからなかった場合は **`⚠️ 辞書に未登録`**（橙色）
* 外部リンクの処理を行った場合は **`ℹ️ 外部リンク補正`**（灰色）
これにより、Markdown側のタイポや、ファイル名の不一致をブラウザ上で一目で検知できる。



### 環境変数による本番制御 (`.env` 連動)

この強力なデバッグパネルは、プロジェクトルートの環境変数ファイル（`.env`）と完全に連動している。

* `PUBLIC_ENABLE_DEBUG=true` の場合、サーバーは検証用HTMLを出力し、フロントエンドのログ出力が有効化される（開発環境用）。
* `PUBLIC_ENABLE_DEBUG=false` に書き換えるだけで、**リンクの自動補正や別タブ化といった強力な裏方ロジックは100%維持したまま、グレーの検証パネルだけを画面上から完全に消滅**させることができる（本番・一般公開環境用）。

同時に、Git管理への安全な共有のために、注意書きのコメントを豊富に記載した `.env.sample` を構築し、チーム開発における安全な環境複製フローを確立した。
