## 🎯 1. 開発の背景と確定している最重要仕様

### ① 住所（URL）の課題を解決する「絶対安全な辞書検索方式」
Markdown内の相対リンク（`../records/ファイル名.md` など）は、日本語URLのエンコード問題や階層の深さによって破綻しやすいため、私たちは「相対パスの計算」を一切排除しました。
* **サーバーサイド（Astro）:** ビルド時に全ファイルの「ファイル名」と「Astroの正規絶対URL」のペア対応表（`urlRegistry`）を自動生成し、HTMLに埋め込みます。（この重い演算はビルド時に1度だけ行われるため、ユーザー閲覧時の負荷はありません）
* **クライアントサイド（ブラウザ）:** JavaScriptがページ内のリンクをスキャンし、この辞書（レジストリ）と合致するファイル名を見つけたら、100%正しい正規の絶対URLへと `href` を強制上書き補正します。これにより、404エラーを完全に根絶しています。

### ② 実装済みのコア機能一覧
1. **ドキュメントツリーのソート:** 同一階層内では、常に「ファイルを先、フォルダを後」の順で並び替えて描画。
2. **サイドバーの初期全閉じ＆URL連動開閉:** 初期表示は全閉じ。ページアクセス時に、ブラウザのURL階層を解析し、該当する親フォルダ（`<details>`）だけを自動展開。
3. **現在地のメニューハイライト:** 表示中ページのサイドバー項目を、青色背景（`bg-blue-50 text-blue-600`）に強調表示。
4. **本文内外部リンクの自動抽出と別タブ化:** Markdown本文内の `http(s)://` リンクを正規表現で自動抽出し、サイドバー最下部に動的にリストアップ。さらに、本文内・サイドバー内を問わず、すべての外部リンクのクリック時に `target="_blank"`（別タブ）を一括注入。
5. **環境変数によるデバッグ制御:** `.env` の `PUBLIC_ENABLE_DEBUG=true/false` によって、画面最上部に表示される「リアルタイム補正ログ検証パネル」の表示・非表示を設定可能。`.env.sample` も整備済。

## 📅 2. 次のフェーズにおける作業予定項目（バックログ）

次のステップとして、以下のタスク（機能拡張や本番化への対応）を予定しています。
※今回開始したいタスク、または新しくやりたいことについては、このプロンプトの後に指示を出します。

* **【タスク1】本文（Markdown）のCSSスタイリング強化・レスポンシブ対応**
  `@tailwindcss/typography`（`prose` クラス）をベースに、社内文書としてより見やすい見出し、テーブルの罫線、コードブロック、引用、警告文（Alert）などの見た目をさらに美しくカスタマイズする。また、スマートフォンやタブレット閲覧時のサイドバーのレスポンシブ対応（ハンバーガーメニュー化）。
* **【タスク2】全文検索（Search）機能の導入**
  Markdown内のテキストやタイトルから、目的の文書をパッと縦断検索できる軽量な検索窓（PagefindやFlexSearchなど）の設計・組み込み。
* **【タスク3】ドキュメントのメタデータ（フロントマター）の拡張とバリデーション**
  各Markdownの先頭に `tags`、`author`、`updatedDate` などを追加し、`src/content/config.ts` の `zod` スキーマを厳格に定義してシステム全体のデータ品質を担保する。
* **【タスク4】静的ビルド（Build）と本番サーバーへのデプロイ準備**
  `npm run build` を実行した際の成果物の検証、ドキュメントルート（サブディレクトリ運用時など）のパス調整、本番環境へのデプロイ手順の確立。
